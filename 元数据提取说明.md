# 元数据提取说明（Zotero → MatrixIt）

本文档基于 [字段表.md](file:///d:/Project/matrix-it/字段表.md) 总结：

1) 为了支持“引用格式/参考文献表”与筛选检索，当前还需要提取哪些 Zotero 元数据字段。  
2) 哪些字段在语义上可以合并（或用统一字段表达类似含义），以减少字段膨胀并保持数据一致性。

---

## 1. 当前 MatrixIt 已提取字段（现状）

当前导入逻辑（[zotero.py](file:///d:/Project/matrix-it/backend/matrixit_backend/zotero.py)）提取的主要元数据：

| MatrixIt 字段 | Zotero 字段/来源 | 备注 |
|---|---|---|
| item_key | items.key | 主键 |
| item_type / type | itemTypes.typeName | 条目类型 |
| date_modified | items.dateModified | 同步增量基础 |
| title | title | 标题 |
| author | creators(author) | 当前是拼接字符串 |
| year | date[:4] | 仅取年份 |
| publications | publicationTitle / proceedingsTitle / bookTitle | 三选一合并 |
| citation | shortTitle | 目前只是占位；不是 GB/T 7714 |
| abstract | abstractNote | 摘要 |
| doi | DOI | DOI |
| url | url | URL |
| collections | collections / collectionItems | 集合信息（含 path） |
| attachments | itemAttachments(storage:) | PDF 附件列表（key + filename） |

结论：**当前字段足够用于“基础列表展示 + PDF 定位 + 分析/同步流程”，但不足以生成可靠的引用格式，也不足以做细粒度检索/筛选。**

## 2. 还需要提取哪些字段（按用途优先级）

### 2.1 为“引用格式”必须补齐的字段（强建议）

这些字段在主流引用样式（GB/T 7714、APA、Chicago 等）中出现概率高，缺失会导致引用不完整：

| Zotero 字段 | 用途 | 典型出现位置 |
|---|---|---|
| date（完整保留） | 年/月/日 | 年份、出版日期 |
| volume | 卷 | 期刊/会议录/图书 |
| issue | 期/号 | 期刊/杂志/部分会议录 |
| pages | 页码 | 期刊/章节/报告等 |
| publisher | 出版社/发布者 | 图书/网页/会议录等 |
| place | 出版地/地点 | 图书/报告/部分会议录 |
| publicationTitle | 期刊名 | 期刊/杂志/报纸 |
| bookTitle | 书名/文集名 | bookSection 等 |
| proceedingsTitle | 会议录名 | conferencePaper |
| conferenceName | 会议名称 | conferencePaper（部分样式显示） |
| websiteTitle | 网站名称 | webpage |
| accessDate | 访问日期 | URL/网页资源常用 |
| edition | 版次 | 图书类常用 |

同时，作者相关建议不再只存 `author` 字符串，至少应补齐结构化 creators：
- creators：`[{creatorType, name|firstName,lastName}]`，保留不同 creatorType（author/editor/translator 等），否则很多条目类型会丢责任者信息。

### 2.2 为“检索/过滤/对齐外部系统”建议补齐（中建议）

| Zotero 字段 | 用途 |
|---|---|
| ISBN / ISSN | 图书/期刊识别与去重 |
| journalAbbreviation | 样式启用时可用于期刊简称 |
| language | 语言过滤 |
| rights | 版权/权限信息（可用于合规提示） |
| extra | 补充信息（很多用户会塞关键信息） |
| citationKey | citekey（与 Better BibTeX 等插件协作；用于 LaTeX 工作流） |

### 2.3 “类型特定”字段（按需）

如果后续要支持更宽的条目类型（专利/法律/视听），需要按类型提取：
- 专利：`patentNumber`、`applicationNumber`、`assignee`、`priorityDate`…
- 法律：`caseName`、`court`、`docketNumber`、`history`…
- 视听：`studio`、`runningTime`、`videoRecordingFormat`…

## 3. 哪些字段可以合并表达（推荐归一化方案）

目标：减少字段数量、降低前端/配置/飞书字段同步成本，同时保留必要细节。

### 3.1 “容器标题”合并：publications 继续，但建议更明确

你们当前的 `publications = publicationTitle|proceedingsTitle|bookTitle` 是正确方向，但建议拆出“容器类型”，避免歧义。

推荐：
- `container_title`：从以下字段按 itemType 决定（或按优先级回退）：
  - 期刊/杂志/报纸：`publicationTitle`
  - 会议论文：`proceedingsTitle`（或补充 `conferenceName`）
  - 章节/文集：`bookTitle`
  - 网页：`websiteTitle`
- `container_type`：`journal|proceedings|book|website|other`

如果暂时不想改字段结构，也可以：
- 继续用 `publications`，但在文档/逻辑里明确“它等价于 container_title”。

### 3.2 “年份”与“日期”合并：year 继续保留，但必须保留原始 date

推荐同时保留：
- `date`：Zotero 原字段（字符串，可能包含年月日/区间/不规范）
- `year`：从 `date` 解析出的 4 位年份（用于排序/过滤）

理由：引用格式往往需要完整日期或至少月份；只存 year 会造成信息不可逆丢失。

### 3.3 “引用”字段合并：不要用 shortTitle 充当 citation

当前 `citation = shortTitle` 会误导业务含义。

推荐分离：
- `shortTitle`：原样保留（用于简称）
- `formatted_citation`：真正渲染出的引用字符串（例如 GB/T 7714），由 CSL 引擎或外部工具生成

如果短期内不引入 CSL 渲染：
- 建议把现有 `citation` 字段改为语义更准确的 `shortTitle` 或 `citation_short`（此变更涉及 fields.json/config.json 与前端表格列，需要单独评估）。

### 3.4 标识符合并：统一为 identifiers

推荐把各种标识符放入一个对象，避免为每个标识符单独加 UI/同步字段：

- `identifiers`：
  - `doi`
  - `isbn`
  - `issn`
  - `pmid`
  - `pmcid`

如果保持平铺字段也可，但建议至少在代码层统一映射入口，避免重复判断。

### 3.5 出版者相关合并：publisher/institution/university 归一为 publisher_name

不同 itemType 会用不同字段表达“发布/出版主体”：
- 图书/网页：publisher
- 报告：institution
- 学位论文：university

推荐：
- `publisher_name`：按 itemType 选择来源（publisher|institution|university）
- 仍可保留原字段（用于精确回写/导出）

## 4. 推荐的“元数据提取结构”（尽量不破坏现有字段）

为了不立刻改动 `fields.json` / UI 表格列 / 飞书字段映射，建议采用“双层结构”：

1) **保留现有顶层字段**（兼容当前 UI/流程）：`title/author/year/type/publications/doi/url/...`
2) **新增一个子对象存“扩展元数据”**（不进入现有表格列，后续再决定要不要上 UI/同步）：

```json
{
  "title": "",
  "author": "",
  "year": "",
  "type": "",
  "publications": "",
  "doi": "",
  "url": "",
  "meta_extra": {
    "date": "",
    "volume": "",
    "issue": "",
    "pages": "",
    "publisher": "",
    "place": "",
    "edition": "",
    "websiteTitle": "",
    "conferenceName": "",
    "proceedingsTitle": "",
    "bookTitle": "",
    "journalAbbreviation": "",
    "ISBN": "",
    "ISSN": "",
    "language": "",
    "rights": "",
    "accessDate": "",
    "extra": "",
    "citationKey": "",
    "creators": []
  }
}
```

说明：
- `meta_extra` 的字段名建议保持 Zotero 原字段名（大小写也保持），减少映射复杂度。
- 顶层字段仅保留你们当前“强依赖”的展示/同步字段，避免牵一发而动全身。

## 5. 提取规则建议（避免脏数据与后续踩坑）

1) **保留原始值 + 派生值**：例如 `date` 原样保留，同时派生 `year`。  
2) **结构化 creators**：至少保留 `creatorType` 与可显示的 `name`，不要只存 author 字符串。  
3) **字段缺失的默认值**：字符串用 `""`，数组用 `[]`，对象用 `{}`，避免前端分支爆炸。  
4) **统一清洗**：去首尾空白、替换不可见字符；不做“过度纠正”（避免改变含义）。  
5) **合并字段要可追溯**：如 `publications` 来自哪个字段，建议在 `meta_extra` 中仍保留原字段。
