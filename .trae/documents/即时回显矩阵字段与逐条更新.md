## 现象与结论
- 你在分析弹窗“原始日志”里仍看不到 Finished 携带的 item/patch，说明前端实际收到的 Finished 事件仍只有 item_key。
- 当前矩阵字段只能在 AllDone 之后触发整库 load_library 才回填；而 load_library 本身会读取/合并 Zotero DB，耗时可达十几秒。
- 要实现“多条分析时，分析完一条，矩阵视图立刻显示一条”，必须让前端在每条 Finished 时拿到该条的最新字段并合并到 state，而不是等待整库刷新。

## 方案选择（更稳、更可维护）
- 不再依赖 sidecar 在 Finished 事件里直接塞完整 item（这在你的实际运行链路里没有生效，且 payload 可能过大）。
- 改为：Finished 到达 → 前端把 item_key 放入队列 → 200ms 内批量向后端请求这些 key 的“最新条目” → merge 到 library.items。
- 后端提供一个轻量命令只读 matrixit.db（不读 Zotero sqlite），从而做到毫秒级返回，避免十几秒延迟。

## 具体改动
### 1) Sidecar：新增轻量命令 get_items
- 在 backend/matrixit_backend/sidecar.py 增加子命令：
  - 输入：JSON 数组 item_keys
  - 输出：{ items: [...] }
  - 数据来源：直接从本地 SQLite（matrixit.db）读取这些 key 的完整条目（包含分析字段）。

### 2) Tauri（Rust）：新增 command get_items
- 在 src-tauri/src/main.rs 增加 tauri::command get_items(item_keys)
- 实现方式：调用 sidecar `get_items`，解析 stdout JSON 并返回给前端。

### 3) 前端：Finished 时批量拉取并 merge
- 在 frontend/src/lib/backend.ts 增加 getItems(itemKeys) 的 RPC 封装。
- 在 frontend/src/ui/hooks/useAnalysisState.ts：
  - 收到 Finished 时将 item_key 推入 ref 队列
  - 用 setTimeout(200ms) 做 batch flush
  - flush 调用 getItems(queuedKeys)，把返回的 items 按 item_key 合并进 library.items（若列表里不存在该条则插入）
  - 这样多条分析时会“完成一条显示一条”，且不会因整库刷新而延迟。

### 4) 兼容与回退
- 保留现有 AllDone 后的 handleRefresh（用于同步收藏夹/元数据）；但即使 refresh 很慢，矩阵字段也已提前回显。
