## 结论（针对你当前“并发=3、非多模态”的测试）
### 1) 为什么 3 篇≈2 分钟仍可能正常
即使并发=3，后端并行链路也存在两个会让总耗时“不像完全并行”的结构性原因：
- **先全量准备、后并发请求**：并行分支 `_analyze_parallel` 会先把所有 PDF 解析完并构造好 tasks，再统一开始 LLM 并发请求；PDF 阶段与 LLM 阶段不能重叠。见 [sidecar.py](file:///d:/Project/matrix-it/backend/matrixit_backend/sidecar.py#L284-L423)。
- **PDF 解析并发受 CPU 限制**：`pdf_workers = min(cpu-1, parallel_count, 6)`，如果机器可用 CPU 核数偏少（例如 2 核）则 PDF 解析基本串行，3 篇的 PDF 阶段会接近线性增长。见 [sidecar.py](file:///d:/Project/matrix-it/backend/matrixit_backend/sidecar.py#L284-L337)。
另外，即使客户端并发=3，LLM 服务端/本地代理也可能对并发做限流或排队（这会让总耗时更接近 2 分钟）。

### 2) 为什么“分析完成但前端仍显示分析中，刷新后才对”
前端行状态只由 `processed_status` 决定；启动时先被乐观写成 `processing`，后续必须靠事件流 `Finished` 把该条置 `done`，或靠结束时刷新从后端回填。见 [useAnalysisState.ts](file:///d:/Project/matrix-it/frontend/src/ui/hooks/useAnalysisState.ts#L29-L109)。
因此只要 `Finished/AllDone` 任一环节偶发丢失/未处理，UI 就可能卡在“分析中”，直到你刷新页面重新从后端加载真实状态。

## 计划改动（最小范围，优先修复 UI 卡状态）
### 1) 前端兜底收口：把 `invoke(start_analysis)` 的返回作为“最终完成信号”
修改 [useAnalysisState.ts](file:///d:/Project/matrix-it/frontend/src/ui/hooks/useAnalysisState.ts)：
- 增加一个本地标记 `didReceiveAllDone`（收到 `AllDone` 时置 true）。
- 在 `await startAnalysisRpc(keys, onEvent)` 返回后做兜底：
  - 若未收到 `AllDone` 或 `analysisInProgressRef.current` 仍为 true，则强制结束分析态（`setAnalysisInProgress(false)` + `analysisInProgressRef.current=false`）并调用 `handleRefresh()`。
- 目标：无论事件流是否完整，sidecar 结束后 UI 必然回填到后端真实状态，不需要用户手动刷新/重启页面。

### 2) Rust 解析稳健性：减少 stdout 行解析导致的 Finished 丢失
修改 [main.rs:start_analysis](file:///d:/Project/matrix-it/src-tauri/src/main.rs#L219-L415)：
- 循环结束后对 `buf` 做一次尾包处理：`buf.trim()` 非空则再尝试按单行 JSON 解析一次（避免最后一行无换行/半包导致事件丢失）。
- 取消逻辑微调：不要在 `recv()` 后立刻因 cancelled break（至少处理完当前已收到的一条 event 再 break），避免“刚收到 finished 就 break”。
