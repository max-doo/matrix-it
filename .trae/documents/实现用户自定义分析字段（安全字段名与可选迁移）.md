## 删除已有字段怎么做（推荐策略）
- 删除字段分两种：
  - **仅删除字段定义（软删除）**：从 `fields.analysis_fields` 移除该 key，并从 UI 列配置（order/visible）里剔除；历史条目里的旧值仍留在本地 JSON/SQLite 中，但 UI/LLM/飞书同步都不会再使用它（等价“隐藏且停用”）。风险低、可回滚（再加回同名 key 即可看到旧数据）。
  - **删除字段定义 + 清理历史数据（硬删除）**：在软删除基础上，额外把所有条目里该 key 的值从本地库中 `pop` 掉；不可恢复，适合确认不用的字段。
- 飞书侧：默认**不主动删除飞书列**（避免误删用户手工数据）；如果字段之前已同步到飞书，会变成“本地不再写入”的遗留列，可选做“重命名为【已废弃】”作为增强。

## 目标
- 允许用户新增/删除/排序/改显示名，且不会因为误改 key 导致历史数据与同步断裂。

## 关键设计（推荐）
- 字段两层：
  - **key（内部ID）**：用于本地存储(JSON 顶层字段)、LLM 输出键集合、飞书 schema 缓存索引；尽量稳定。
  - **name（显示名）**：UI 展示与飞书字段名；允许随时改。

## 方案 A（最小改动，优先落地）
1. **新增字段**：保持现有流程（输入 key/name/描述/模式）。
2. **修改字段**：
   - 保存后锁定 key，不允许直接编辑；只允许改 name/描述/模式。
3. **删除字段**：删除按钮弹窗提供两项：
   - 默认：仅删除字段定义（软删除）。
   - 可选：勾选“同时清理历史数据（不可恢复）”执行硬删除。

## 方案 B（可选增强：允许改 key 并自动迁移）
1. 新增后端命令（Tauri Command）：
   - `rename_analysis_field_key(oldKey, newKey, mode)`：迁移条目字段值 + 更新 config 引用 + 更新飞书 schema 缓存。
   - `purge_item_field(key)`：硬删除时清理所有条目的该 key。
2. 迁移/删除涉及的配置点：
   - `config.fields.analysis_fields`（对象键重命名/删除）
   - `config.ui.table_columns.matrix.analysis.order/visible`（剔除或替换 key）
   - `config.feishu.schema.fields`（迁移/删除对应缓存，避免飞书字段绑定错位）

## 验证方式
- 前端：lint/build；手动验证新增字段、删除（软/硬）后表格/详情页行为。
- 后端（若做方案 B）：构造含 oldKey 的条目，验证软删除不再显示、硬删除确实清掉、重命名后历史数据与飞书映射不断。

## 默认交付范围
- 默认先做 **方案 A（安全删除 + key 锁定）**。
- 若你确认需要“改 key 自动迁移/硬删除全库清理”，再扩展实现方案 B。
