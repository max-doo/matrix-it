## 原因分析
- 报错来源：校验同步状态会调用 [batch_get_records](file:///d:/Project/matrix-it/backend/matrixit_backend/feishu.py#L877-L916)，请求 `app_token/table_id` 对应的数据表；当接口返回 `success=false` 时直接抛出 `batch_get failed: code=... msg=...`。
- `code=1254302 msg=RolePermNotAllow` 的含义：当前应用（由 `app_id/app_secret` 代表）对“新的多维表格/数据表”没有足够权限（典型场景：未把应用加为该 Base 协作者 / Base 开启高级权限导致应用角色无权读写）。因此 batch_get 直接被拒绝，而不是返回“absent_record_ids”。
- 现有逻辑问题：更换 Bitable URL 后，本地 item 仍携带旧表的 `record_id` 且 `sync_status==synced`，启动/点击“更新”会继续对这些旧 record_id 做校验（在新的表里没有意义）。这也是你期望“应当根据新的表格检查同步状态”的核心。

## 目标行为
- 更换 Bitable URL（app_token/table_id）后：本地所有条目相对于“新表”应视为未同步；校验逻辑不再对旧表的 record_id 发 batch_get。
- 若新表确实没给应用权限：错误提示更明确（提示去飞书把应用加协作者/授予编辑权限），避免误判为“程序 bug”。

## 实施方案（代码改动）
### 1) 引入“表签名”并在条目中记录
- 在同步成功时，把当前 `app_token/table_id` 写入每个 item（例如 `feishu_app_token`、`feishu_table_id` 或 `feishu_base_sig`）。
- 在校验（reconcile）时，仅对“条目记录的表签名 == 当前配置表签名”的条目做 batch_get；若不匹配：清除 `record_id`、置 `sync_status=unsynced`。

### 2) 配置 schema 缓存与表签名绑定
- 目前 `feishu.schema.fields` 会缓存 field_id；更换表后继续沿用可能导致后续字段 rename/类型校验异常。
- 方案：在 config 里把 schema 缓存附带 `schema.base_sig`；当检测到 base_sig 变化，忽略/清空旧 schema 缓存并重新拉取/创建。

### 3) 改善错误提示
- 当捕获到 `RolePermNotAllow`：在前端 toast 或错误文案中提示“新多维表格未授予应用权限/高级权限限制”，并引导去飞书把应用添加为协作者或授予编辑权限。

## 验证方式
- 用一个全新表（未建字段）首次同步：字段创建顺序按既定顺序；并写入条目的表签名。
- 改 Bitable URL 到另一个表后：启动自动校验不再报错；条目被标记为待同步；点击同步走新表流程。
- 将新表权限刻意设置为无权：仍会报错但提示为权限问题（RolePermNotAllow），符合预期。

## 交付物
- 后端：reconcile/sync 对表签名不匹配的条目自动“断开旧绑定”。
- 前端：权限错误提示更可操作。
