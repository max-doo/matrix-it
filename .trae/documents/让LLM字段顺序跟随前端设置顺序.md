## 评估：在 fields.analysis_fields 中加 id 决定顺序，是否合理？
- **可行**，而且比依赖 JSON 对象的“物理键顺序”更稳：JSON 对象本身不保证业务顺序，很多读写/合并会改变键排列；把顺序信息写进字段定义里（每个字段一个数值）是常见做法。
- **但不建议用“id”承载顺序语义**：
  - 你现在的 `analysis_fields` 的 key（比如 `methods`）本身已经是稳定标识符，拿来做“id”更合理；再加一个 `id` 容易跟“身份标识”混淆。
  - 顺序更推荐叫 `order` / `position` / `rank`（整数或可比较值），语义清晰。
- **优点**：
  - 顺序是数据模型的一部分：无论前端如何保存、后端如何读取，排序都可复现。
  - 不需要额外依赖 UI 配置字段（比如 `ui.table_columns...order`），避免“显示顺序/推理顺序”两套来源打架。
  - 便于扩展：未来可以按 `order` 分段（先核心字段、再扩展字段），或做 A/B 分组内排序。
- **风险/成本**：
  - 需要一次性做兼容：旧配置没有 `order` 时要有回退策略（按现有 keys 顺序/或按设置页当前顺序生成）。
  - 需要保证唯一性/稳定性：若用 `id` 作为顺序且用户可编辑，可能出现重复、非数字、删除导致“空洞”等问题；需要校验与自动修复。
  - 如果你还希望“表格列顺序”和“LLM 顺序”可分别配置，字段内 `order` 会更偏向“LLM/语义顺序”，表格仍需自己的列顺序配置（这两者往往确实不同）。

## 推荐设计（兼顾可用性与清晰语义）
- **在每个字段定义里新增 `order`（整数）**，用它决定 prompt 的 keys 顺序。
- `analysis_fields` 的 key 仍是字段标识，不改现有结构。
- 顺序策略可选：
  - 默认：按 `order` 升序（缺失的放最后，并按 key 字母排序兜底）。
  - 可选增强：先规则A后规则B（组内仍按 `order`），避免把“需要引用的字段”挤到后面。

## 落地计划（需要改代码时我将按此执行）
1) **前端设置页**：在“解析字段”的拖拽/上下移动/显示列表顺序变化时，同步更新每个字段的 `order`（例如从 10 开始递增或按 index 递增）。并在保存时写回 `config.fields.analysis_fields[<key>].order`。
2) **后端 prompt_builder**：生成 keys 时不再依赖字典插入顺序；改为优先按字段定义的 `order` 排序（再用 key 做稳定兜底），然后再生成 rule_a_fields/rule_b_fields 与 analysis_fields JSON 文本。
3) **兼容迁移**：读取到没有 `order` 的旧配置时：
   - 以当前字段列表顺序（或当前 keys 顺序）自动补齐 `order`；
   - 不影响旧配置立即可用。
4) **验证**：调整设置页顺序 → 触发一次分析 → 观察模型生成字段的顺序/完整度是否随 `order` 变化，并确认写库与 UI 展示正常。

## 验收标准
- 只要 `config.fields.analysis_fields.*.order` 不变，发送给 LLM 的字段 keys 顺序永远一致，不再“突然回到之前的顺序”。
- 调整 `order` 后，LLM 输出顺序/完整度变化可复现。
