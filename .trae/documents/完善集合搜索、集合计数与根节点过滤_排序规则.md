## 新增要求补充

* 根节点“我的文献库”不能被折叠：始终保持展开状态、也不显示可折叠的 switcher。

* 需要评估并规避：按“集合名 + 集合内条目标题”搜索可能带来的卡顿/高延迟。

## 目标与现状

* 现状：侧边栏集合搜索仅做 `indexOf` 高亮，不过滤树；集合节点不展示条目数；根节点选中等价于 `activeCollectionKey=null`，条目列表不做类型过滤；标题升序排序空标题排最前。

* 目标：

  * 集合树搜索支持模糊匹配集合名称 + 该集合（含子集合）下条目标题；命中集合高亮；根节点不参与检索。

  * 每个集合显示该集合拥有的条目数量。

  * 根节点视图过滤掉 `annotation`；标题升序时无标题排最后。

## 方案概述

### 1) 根节点不可折叠（强制常开）

* 树结构仍以根节点包裹所有集合，但做两层防护：

  * UI 层：对根节点设置 `switcherIcon: () => null`（不显示展开/折叠箭头），避免用户有折叠入口。

  * 逻辑层：`onExpand` 时强制把根 key 加回 `expandedKeys`（即使用户通过键盘/内部行为触发折叠，最终也会保持展开）。

* 选中逻辑不变：点击根节点回传 `null` 表示“全库”。

### 2) 侧边栏：集合模糊搜索（集合名 + 条目标题）+ 命中高亮（根节点不参与）

* 输入：给 `AppSidebar` 传入 `library.items`（或传入预计算索引）。

* 计算：基于当前搜索词生成 `matchedCollectionKeys`：

  * **集合名命中**：对每个集合节点名做“子序列模糊匹配”（不引新依赖）。命中则加入其 key。

  * **条目标题命中**：扫描条目标题命中搜索词的条目，对其 `collections[].key` 及 `pathKeyChain` 中祖先 key 加入 `matchedCollectionKeys`（去重）。

* 树过滤：搜索词非空时，只保留“自身命中或任一后代命中”的集合节点；根节点始终保留且**不做命中判断**。

* 高亮：对 `matchedCollectionKeys` 中的集合标题整体加高亮样式（不做逐字高亮，避免模糊匹配下复杂的字符映射）。

### 3) 侧边栏：集合条目数量

* 计算 `countByCollectionKey`：对每个条目收集其所属集合 key 及祖先 key（来自 `pathKeyChain`），对这组 key 去重后分别计数 +1，避免同一条目因多路径重复累计父集合。

* 渲染：集合标题显示  `集合名   count`（count对齐容器右侧，保持现有视觉风格，改动最小）。

### 4) 根节点过滤 annotation

* 在 `useCollectionItems` 的 `activeCollectionKey === null` 分支过滤掉 `annotation`：通过读取 `it.type ?? it.item_type` 判断。

* 同步类型：在 `types.ts` 给 `LiteratureItem` 补充 `type?: string; item_type?: string`，减少 `Record<string, unknown>` 的不安全访问。

### 5) 标题升序：空标题排最后

* 在 `LiteratureTable` 两处排序同时改：

  * 标题列 `sorter`：当升序时，空标题视为最大值排最后。

  * `sortedData` 手动排序：当 `sortState.key===TITLE_KEY && order==='ascend'` 时应用同样规则，保证默认排序与点击排序一致。

## 性能评估与优化（针对“按条目标题搜索”可能卡顿）

### 复杂度预估

* 每次输入触发：

  * 条目扫描：O(N\_items × 平均标题长度) 做模糊匹配。

  * 集合扫描：O(N\_collections × 平均集合名长度)。

* 当条目规模达到数千\~上万时，逐键同步扫描可能导致输入卡顿。

### 规避方案（不引新依赖）

* **输入去抖/延迟计算**：

  * 使用 React `useDeferredValue(searchValue)` 作为真正参与匹配的 query，让输入保持流畅，匹配结果稍后更新。

  * 或者对 `onChange` 做 150–250ms 的 debounce（用 `setTimeout` 清理实现，不新增库）。

* **预索引以减少重复工作**：

  * 对 `library.items` 变更时，预先构建 `itemIndex`：`[{ titleNorm, collectionKeys:Set<string> }]`，把“归一化 title + 收集所属集合 key/祖先 key”从每次按键移动到 items 变化时执行。

  * 搜索时只做“遍历 titleNorm + fuzzyMatch(query)”并把命中条目的 `collectionKeys` 合并进 `matchedCollectionKeys`，避免反复处理 `collections/pathKeyChain`。

* **短路策略**：

  * query 为空直接返回完整树；query 很短（例如 1 个字符）时仍可匹配，但 fuzzyMatch 会快速失败/成功，且 `useDeferredValue` 已能明显降低主线程压力。

* **可选增强（如需要）**：若仍有卡顿，可在后续迭代引入 Web Worker 做搜索，但这会涉及新增文件/配置（届时再单独征求你同意）。
